# 阶段 6：行动与重大事件记忆

状态：完成并通过真实游戏验证。

## 本阶段目标

- 让复制人记得自己和同伴真正完成过的工作。
- 让所有复制人知道科技完成和成员死亡等重大殖民地事实。
- 严格区分“已经发生”和“准备去做”，避免虚假行动记忆。
- 不为记忆提取增加额外 AI 请求。

## 捕获语义

当前游戏 DLL 已确认提供 `Workable.CompleteWork(WorkerBase)`。ONITalk 在调用前提取仍然有效的工作对象和执行者，在方法正常返回后才提交记忆。因此取消、打断和失败任务不会进入记忆。

行动白名单包括：

- 建造、修理、挖掘和拆除；
- 收获、制作和研究；
- 治疗、生物照料、设备调校和探索。

如厕、睡觉、开门、娱乐和普通设备操作默认忽略。

## 记忆结构

- 每条行动包含执行者、分类、目标、首次/最近周期、次数和重要性。
- 默认每位复制人最多保存 30 条行动。
- 默认在 2 个周期内重复的相同工作合并计数。
- 重大事件单独保存，死亡等最高重要性事件在容量裁剪时优先保留。
- 提示词默认注入双方各 3 条行动和 3 条殖民地事件。

## 持久化迁移

殖民地记忆文件升级为 schema 2，新增 `actions` 和 `events`。读取器继续接受 schema 1；旧文件无需用户手动处理，下一次保存时自动升级。

## 自动验证

- Core 检查：40/40 通过。
- Release 构建：0 警告、0 错误。
- 覆盖：重复聚合、跨窗口新事件、个人容量隔离、导入导出、重大事件去重、关键事件保留、行动分类白名单与噪声过滤、提示词注入。

## 游戏内验收清单

1. 让复制人完成一次挖掘、建造或制作。
2. 日志出现 `Action memory recorded`，执行者与目标正确。
3. 等待保存周期后，殖民地 JSON 升级为 schema 2 并出现 `actions`。
4. 同一复制人重复完成相同工作，记录合并并增加 `count`。
5. 后续对话正常生成，且没有 Harmony、记忆或 API 错误。

## 首轮实测结果

- schema 2 已成功写入，当前殖民地记录 9 条个人行动。
- 建造电线与挖掘均正确聚合，计数分别增长到 6 次。
- 后续台词能够自然引用“铺了六次电线”等事实，证明行动记忆已进入提示词。
- 首轮测试发现睡眠中的复制人仍会被环境扫描选中；修正版增加 `Asleep`、`Dead`、`Dying`、`Incapacitated`、`Entombed` 和 `Stored` 状态过滤，并已通过游戏内复测。
- 修正版运行期间未发现 Harmony、记忆、API 或聊天窗口错误。
