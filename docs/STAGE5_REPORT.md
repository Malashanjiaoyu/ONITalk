# 阶段 5：双向连续对话

状态：完成并通过真实游戏验证。

## 本阶段目标

- 让听者真正回应上一句，而不是继续生成互不相关的独立台词。
- 保持“只生成文本，不控制游戏行为”的边界。
- 明确限制轮次、并发和中断条件，避免请求风暴。

## 会话流程

1. 游戏事件选择说话者与听者，并创建唯一会话。
2. 后台生成第一句，主线程将其显示为头顶气泡并加入聊天窗口。
3. 主线程等待配置的回应间隔。
4. 确认双方仍存在且距离合适，重新抓取回复者与听者状态。
5. 将对方刚说的话和长期关系记忆一起送入下一轮提示词。
6. 达到台词上限后释放会话，下一次触发才可开始。

## 配置

- `conversationRepliesEnabled`：是否允许听者回应，默认开启。
- `replyChancePercent`：回应概率，默认 100%。
- `maxConversationLines`：一次会话总台词数，范围 1 至 3，默认 2。
- `replyDelaySeconds`：两句之间的等待时间，范围 0.5 至 5 秒，默认 1.5 秒。

## 安全与成本控制

- 同时只允许一个活动会话；会话内部 API 请求严格串行。
- 距离过远、参与者失效、存档切换或服务重载会终止后续轮次。
- 会话使用存档记忆世代号，旧存档迟到的结果不会进入新存档。
- DeepSeek 不可用时使用可回应的离线台词，游戏模拟不会被阻塞。

## 自动验证

## 游戏内验收清单

1. 日志显示 `Dialogue session started` 且 `targetLines=2`。
2. 第一位复制人发言后，第二位复制人在设定间隔后回应。
3. 回应内容与第一句有关，不是简单复述。
4. 日志显示 `Dialogue session completed` 且 `lines=2`。
5. 聊天窗口按正确顺序显示双方台词，长期记忆累计增加 2。

## 实测结果

- 日志连续出现 `Dialogue session started`、两位复制人的台词和 `Dialogue session completed`。
- 多组会话均使用 `targetLines=2`，结束时报告 `lines=2`。
- 听者能够直接承接上一句中的漏电、气体和设备话题，没有退化为无关独白。
- 头顶气泡与聊天历史窗口均按正确顺序显示。
- 会话期间未发现 ONITalk、DeepSeek、长期记忆或聊天窗口错误。
